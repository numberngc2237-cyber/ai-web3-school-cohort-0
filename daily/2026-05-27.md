### 关键概念

<!-- 用自己的话记录 2-3 个核心概念 -->

1. **智能合约 = 链上公开规则，不是"后端代码"**
   合约一旦部署到链上，所有人都能看到它的逻辑和状态。前端可以换、后端可以重构，但合约改起来代价极大。这跟传统互联网产品完全不一样——我们习惯了"先上线再改"，但合约的世界是"部署前必须想清楚"。这对产品经理来说意味着：需求评审的严谨度要比传统产品高一个量级。

2. **ABI 是接口说明书，不是安全说明书**
   ABI 告诉前端/AI Agent "能调用哪些函数、参数是什么类型"，但它不告诉你：这个地址是不是恶意合约、amount 的单位是不是正确的 decimals、调用会不会触发未知逻辑。对 AI Agent 来说尤其危险——Agent 看到 ABI 会以为"我会用了"，但实际上安全性需要额外的验证层。这让我理解了为什么 AI Agent 调用合约时，钱包的"人工确认"环节不能省略。

3. **Event 是合约的"日志系统"，产品页面的数据来源**
   产品页面通常不直接扫链上 storage（太贵且慢），而是通过监听合约发出的事件（Event）来构建前端数据。比如 NFT 市场的订单列表、交易历史、用户看板都来自事件索引。这意味着合约的事件设计直接影响产品的可观测性——设计得不好，用户看不到自己的操作历史，产品体验就崩了。

### 产品视角

<!-- 从产品研究方向思考：这个概念对应什么产品场景？用户是谁？解决什么问题？ -->

| 概念 | 产品场景 | 用户是谁 | 解决什么问题 |
|------|---------|---------|------------|
| **智能合约** | DeFi 借贷协议（如 Aave） | 想赚利息的持币者 + 需要借钱的交易者 | 替代银行中介——双方不需要信任对方，规则由合约保证执行 |
| **EVM + Gas** | 任何链上操作都需要付费 | 所有用户 | 防止无限循环和垃圾交易——Gas 是"计算资源"的定价机制，类似云服务按量付费 |
| **ABI** | AI Agent 自动调用合约 | Agent 开发者 + 最终用户 | Agent 需要"读懂"合约才能调用——但 ABI 不保安全，所以 Agent 产品必须加人工确认环节 |
| **Event** | NFT 市场交易历史页 | 买家、卖家、数据分析者 | 用户需要看到"我的订单状态变了"——Event 是前端显示的数据源 |
| **可升级 vs 不可升级** | 协议治理（如 Uniswap） | 协议团队 + 持币用户 | 不可升级 = 规则固定但信任度高；可升级 = 能修 bug 但引入了"谁有权限改"的治理问题 |
| **权限控制（modifier/owner）** | Token 合约的 mint 权限 | 项目方 + 用户 | 谁能增发 token？权限太集中 → 用户风险；权限太分散 → 无法迭代 |

**核心产品洞察**：智能合约把"信任"从人转移到了代码。传统产品靠品牌和合同建立信任（"相信淘宝不会吞我的钱"），Web3 产品靠合约的公开可验证性建立信任（"任何人都能验证这个借贷协议不会卷款跑路"）。但代价是——如果代码有 bug，信任瞬间崩塌，且修复成本极高。所以 Web3 产品经理的核心能力之一是：判断什么逻辑必须上链、什么逻辑可以放链下。

## 问题和卡点

<!-- 记录今天遇到的困惑、不确定的地方 -->

1. **Gas 费的定价逻辑**：理解了 Gas 是计算资源成本，但 Gas 价格为什么会波动？用户什么时候交易便宜？这个波动对产品体验的影响有多大？（比如 DeFi 产品在 Gas 高峰期用户流失率会上升多少？）

2. **代理升级模式的信任问题**：如果合约是可升级的（proxy 模式），用户怎么知道新版本有没有改规则？有没有类似 App Store 更新日志的"合约变更通知"机制？产品侧应该怎么设计这个透明度？

3. **AI Agent 调用合约的安全边界**：Handbook 说 AI 做建议和编排 → 钱包做授权确认 → 合约做可验证执行。但如果是高频交易场景（AI Agent 自动调仓），不可能每笔都人工确认——那安全边界应该设在哪里？金额阈值？白名单合约？时间窗口？
