### 关键概念

用自己的话记录今天学到的 3 个核心概念：

1. **RAG 不是「给模型接个数据库」，而是一条证据链**  
   检索 → 筛选 → 引用 → 交给模型用。核心目标是让回答**可验证**——每个关键判断都要能追溯到具体来源、版本和时间。不是让 AI 说得更多，而是让 AI 说得有依据。

2. **Agent = LLM + 工具 + 多步推理循环**  
   Agent 和普通 AI 聊天不一样：它不是单轮回答，而是「拆任务 → 查资料 → 调 API → 生成操作 → 等反馈 → 继续下一步」的循环。关键不在「像人」，而在「能不能在明确的权限和可审计的流程里把建议推进到行动」。

3. **Agent 的三层安全边界：模型提议 → 系统限制 → 用户确认**  
   模型负责提出候选行动（但没权力执行），系统负责限制行动空间（只读/写入分开），用户负责批准高风险边界。最危险的设计：Agent 同时拥有模糊目标 + 广泛工具 + 长期记忆 + 大额资产权限。

### 产品视角

> 从产品研究方向思考：这个概念对应什么产品场景？用户是谁？解决什么问题？

| 概念 | 产品场景 | 目标用户 | 解决什么问题 | 关键设计约束 |
|------|---------|---------|-------------|------------|
| **RAG** | 协议文档智能问答、治理提案摘要、审计报告检索、SDK Copilot | 开发者、研究者、DAO 成员、审计人员 | 「这个函数干嘛的？」「这个提案改了啥？」→ 让 AI 给出有出处、有版本、可核对的答案 | 引用必须能回到原文；旧版本文档可能相似但已不适用；需要区分「能检索到的」和「是正确的」 |
| **Agent（只读型）** | DAO 提案研究 Agent：读提案 → 检索讨论 → 总结正反理由 → 标记缺失信息 → 输出投票前检查清单 | DAO 成员、治理参与者 | 面对几十页提案 + 几百条讨论，一个人根本看不过来。Agent 帮你把关键信息摘要出来，不带偏见地汇总正反观点 | 只读不写，风险可控；但总结的偏见需要警惕——模型倾向于「写得好」而非「判断对」 |
| **Agent（写入型）** | DeFi 仓位管理 Agent：监听市场 → 分析风险 → 生成调整建议 → 走 simulation → 用户确认 → 签名执行 | 普通投资者、DeFi 用户 | 手动调仓太慢，但又不敢完全交给机器人。Agent 做分析 + 提议，人做最终确认 | 必须过 simulation + policy 检查 + 人工确认三层防线；工具必须区分「查看余额」和「转移资产」；停止条件必须明确 |

**Phase 1 产品洞察总结**：

从 LLM → Prompt → Context → RAG → Agent，这五个概念构成了一个产品能力堆栈：
- **LLM** 是「引擎」——决定 AI 能做什么
- **Prompt** 是「方向盘」——让人告诉 AI 往哪开
- **Context** 是「视野」——AI 一次能看多少
- **RAG** 是「知识库」——让 AI 回答有据可查
- **Agent** 是「自动驾驶」——让 AI 从回答问题变成执行任务

对于产品研究者来说，最重要的是理解每个层的**责任边界**和**失败模式**——什么情况下会出问题、出问题后用户怎么发现、怎么兜底。

### 问题和卡点

1. **RAG 和微调到底怎么选？**  
   两者都让模型获取外部知识，但用起来完全不一样。RAG 适合知识频繁更新的场景（协议文档、新闻），微调适合需要风格化输出的场景。希望 Handbook 能加一个对比表——适用场景、成本、更新频率一目了然。

2. **Agent 的「停止条件」怎么设计？**  
   Handbook 说 Agent 要设定停止条件（达到目标、超出预算、信息不足、风险越界）。但实际产品里，谁来定义「什么算信息不足」？如果 Agent 自己判断「信息够了」但其实不够呢？这个边界感觉还很模糊。

3. **多 Agent 协作到底值不值得？**  
   Handbook 提了多 Agent 会放大协调问题。对于我现在理解的水平，感觉单 Agent + 清晰工具边界可能比多 Agent 更实用？不知道实际项目里什么规模才需要多 Agent。
