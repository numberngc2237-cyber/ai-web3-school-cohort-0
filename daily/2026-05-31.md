# 2026-05-31 学习笔记

## 今日主题

**🚀 Phase 3：AI × Web3 Bridge — Day 2**：Web3 工具调用（Web3 Tool Use）

> 昨天完成了链感知上下文（3.1）——理解了 AI Agent 不能凭记忆判断链上事实，链上状态有时间性、必须带来源、要区分事实和解释。今天进入 Web3 工具调用（3.2），学习如何把 RPC 调用、合约读取、交易生成、钱包确认等操作封装成 AI Agent 可以安全调用的结构化工具——这是从「AI 能理解链上信息」走向「AI 能参与链上执行」的关键一步。

## 今日学习路径

> 根据你的画像（AI新手 / Web3新手 / 无代码 / 产品研究方向 / 每日2h），选一条开始：

### 🟢 最小路径（~30min）

先读「为什么要学」+「第一性原理」+ RPC Tool + Contract Read + Explorer Tool 三个初级知识节点。

- 📖 [Web3 工具调用 Web3 Tool Use](https://aiweb3.school/zh/handbook/bridge/web3-tool-use/) — 为什么要学 + 第一性原理
- 🎯 重点理解三个基石原则：**读写分离**（读链上状态和发交易是不同的工具、不同权限）、**参数结构化**（chain_id / contract_address / method / args 不能埋在自然语言里）、**日志不可省**（每次调用必须记录输入输出时间来源错误）
- 💡 产品思考：为什么 Web3 工具的风险等级远高于普通查询工具？读错会误导判断，写错会直接改变资产和权限——这对产品设计意味着什么？

### 🔵 推荐路径（~1h）

完成最小路径后，继续读 Contract Write（高级）+ Wallet Tool（高级）+ Tool Log（中级）+「在 AI × Web3 中的位置」。

- 📖 同上链接 — Contract Write（8 项检查清单）+ Wallet Tool（最敏感边界）+ Tool Log + AI × Web3 中的位置
- 🎯 重点理解：Contract Write 必经 8 项检查——chain_id、ABI method、value 预估、gas 估算、simulation、policy 检查、授权确认、交易哈希追踪。Wallet Tool 必须拆分连接/签名/发送/授权/撤销，AI 生成的交易草稿不能绕过钱包确认
- 💡 产品思考：假设你设计一个 AI 链上助手产品，L0（自动允许查询）→ L4（默认禁止任意合约调用）这五级权限各应该对应什么用户场景？

### 🟣 挑战路径（~2h）

完成推荐路径后，读 DeFi Tool（高级）+ Tool Permission（L0~L4 五级权限模型全解）+ 做「最小实践」+ 阅读扩展资料。

- 📖 同上链接 — DeFi Tool（7 项约束）+ Tool Permission 完整五级模型 + 最小实践
- 🔬 最小实践（5 步产品设计，无需代码）：① 设计读取某地址 ETH 余额的工具 schema → ② 设计读取 ERC-20 allowance 的工具 → ③ 设计生成 ERC-20 approve calldata 但不发送交易的工具 → ④ 定义只允许特定 token、特定 spender、最大额度的写交易权限规则 → ⑤ 为每个工具定义输入 schema、输出字段、错误类型和日志字段。**重点不是实现，而是把读、写、签名、确认、记录五个边界分清楚**
- 📚 扩展阅读：[OpenAI Tools Guide](https://platform.openai.com/docs/guides/function-calling)、[Ethereum JSON-RPC API](https://ethereum.org/en/developers/docs/apis/json-rpc/)、[viem Documentation](https://viem.sh/)、[Tenderly Simulations](https://tenderly.co/)
- ✍️ 产品分析草稿：选一个 AI × Web3 产品（如 TruthTerminal、virtuals.io 等 AI Agent 项目），分析它的工具调用机制——它允许 Agent 调用哪些工具？权限边界在哪？日志和审计能力如何？
