### 关键概念

用自己的话记录今天学到的 3 个核心概念：

1. **区块（Block）= 一页不可篡改的账本**
   每个区块像一个盖了时间戳的「账本页」，里面记录了多笔交易。改一页就要重算后面所有页的哈希——成本极高，这就是「不可篡改」的真正含义。对产品来说：用户看到的「已确认」背后是一整条链的保护，不是某一个节点的承诺。

2. **共识（Consensus）= 大家一起决定哪页账本有效**
   区块链不是靠一个中心化服务器说「这页有效」，而是全网节点按规则（PoW/PoS）各自检查，最终多数人认可的那页才写入。核心是「多数人诚实」的前提假设。对产品来说：这就是为什么交易需要等 N 个确认——等的时间越长，被推翻的概率越低。

3. **L2 和 Rollup = 在高速辅路上跑，定期回主路结算**
   以太坊主网（L1）慢且贵，L2 把大部分计算和交易搬到「辅路」上，只定期把结果打包回主路（Rollup）。这就像你不是每笔交易都去银行排队，而是在咖啡店用会员卡记账，月底统一结算。对产品来说：L2 能让用户用更便宜的价格交互，但跨 L2 转账/查余额需要额外设计——用户看到的不再是一条链上的状态。

### 产品视角

从产品研究方向思考：这个概念对应什么产品场景？用户是谁？解决什么问题？

| 概念 | 产品场景 | 目标用户 | 解决什么问题 | 关键设计约束 |
|------|---------|---------|-------------|------------|
| **Block + Confirm** | MetaMask 钱包「等待确认」UX：用户发起转账后看到 pending → confirmed | 普通 Web3 用户 | 「我的钱到没到？」→ 用区块确认数代替人工客服验证 | 确认数多少合适？太少不安全，太多用户等太久。不同链、不同金额可能需要不同策略 |
| **L2 / Rollup** | 多链资产仪表盘：一个界面看到用户在 Ethereum + Arbitrum + Optimism 上所有资产 | DeFi 用户、多链玩家 | 用户不想切换 RPC/网络才能看到自己所有链上的钱 | L2 数据获取需要额外的索引服务；不同 L2 的确认时间和 finality 不同，UI 需要标注状态 |
| **Testnet** | 新人学习钱包操作：在 Sepolia 测试网免费领 ETH → 体验转账、合约交互 | Web3 新手 | 怕输错地址、怕丢真金白银 → 在测试环境安全练习 | 测试币「不值钱」的心理暗示可能让用户不够认真；需要设计成「和主网体验完全一致但零成本」 |

**Phase 2 第一天产品洞察**：

从产品视角看 Web3「网络」层，最核心的一个认知转变是：**互联网的产品是「假设网络无限快」，Web3 的产品是「假设网络不可信」**。

- 互联网：用户点按钮 → 服务器立刻返回结果 → 用户觉得「快就是好」
- Web3：用户点按钮 → 交易进 mempool 排队 → 等待验证 → 区块确认 → 用户觉得「慢但安全」

这个时间差决定了 Web3 产品的核心 UX 挑战：**如何让用户理解并接受「延迟 = 安全」这个反直觉等式**。互联网 20 年把用户培训成了「加载超过 3 秒就关掉」，Web3 产品现在要做的是让用户觉得「等待 12 秒（一个区块）是值得的」。这不是技术问题，是认知教育问题——而好产品可以把这种教育藏进体验里。

## 问题和卡点

记录今天遇到的困惑、不确定的地方：

1. **「去中心化」到底去掉了谁的中心？**
   Handbook 说区块链是去中心化网络，但实际操作中：RPC 节点（Infura/Alchemy 占了大量流量）、浏览器（Etherscan 是中心化服务）、钱包（MetaMask 连接 Infura 默认）。用户真正接触到的入口还是中心化的——这算不算「入口中心化、底层去中心化」？产品设计时，入口层的单点故障风险怎么让用户感知到？

2. **不同 L2 之间到底能不能互操作？**
   读了 Network 章节后，感觉 L2 像是各自独立的「省道」。同一个 L2 内转账很快很便宜，但 Optimism 上的资产转去 Arbitrum 需要走跨链桥——又回到 L1 的速度和费用。对于不懂技术的新手，这会不会是更大的认知负担？「你钱包里的 ETH 不是同一种 ETH」这种解释对产品来说太难了。
