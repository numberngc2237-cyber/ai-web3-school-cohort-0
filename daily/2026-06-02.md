### 关键概念

1. **框架是 AI 应用的「脚手架」——降低从 0 到 1 的成本**：直接调 API 就像用砖头和水泥从零盖房子——每一块都要自己搬。框架提供了预制好的「提示管理、链式调用、工具集成、记忆管理」模块，让你专注于产品逻辑而非底层实现。对产品人来说：理解框架的能力边界 = 理解你的产品「能多快做出来」和「能做多复杂」。

2. **四种框架 = 四种产品架构范式**：LangChain 是「流水线」——串行链式调用，适合问答、摘要等线性任务；LangGraph 是「流程图」——状态图编排，适合多步 Agent 工作流（先查数据→再判断→再执行）；CrewAI 是「团队协作」——多 Agent 角色分工，适合复杂任务需要「不同专长 Agent 各司其职」的场景；Hermes 是「Web3 原生工具箱」——链上工具原生集成，适合做链上数据分析、交易助手等 Web3 垂直产品。

3. **选框架 = 选技术路线——影响产品形态和迭代速度**：框架不只是工程选择，更是产品约束。选了 LangChain = 你走通用 LLM 应用路线，生态最大但 Web3 能力全靠自己拼；选了 Hermes = 你走 Web3 垂直路线，链上能力开箱即用但生态更小。产品人不需要会写框架代码，但必须能判断「这个产品需求用哪个框架最快落地」。

### 产品视角

| 概念 | 产品场景 | 用户是谁 | 解决什么问题 |
|------|---------|---------|------------|
| **LangChain（链式调用）** | 智能客服、文档问答、内容生成 Pipeline | 企业客户、内容运营团队 | 把多个 AI 任务串起来——「先总结文档→再提取关键信息→再生成报告」，减少人工串联成本 |
| **LangGraph（状态图编排）** | 多步 Agent 工作流、自动化决策链路 | 需要复杂 AI 自动化的业务团队 | AI 不只是「一问一答」，而是「根据上一步结果决定下一步做什么」——适合需要条件判断和多分支的产品 |
| **CrewAI（多 Agent 协作）** | AI 研究团队、内容生产流水线 | 中小团队、创业公司 | 让多个 AI Agent 像团队一样协作——「研究员 Agent 查资料 → 分析师 Agent 出报告 → 编辑 Agent 润色」，每个 Agent 专注一件事 |
| **Hermes（Web3 原生）** | 链上 AI 交易助手、DeFi 智能顾问、链上数据分析 | Web3 用户、DApp 开发者 | 直接读链上数据、调合约——不需要「先找 RPC → 再写调用代码」，链上能力开箱即用 |

## 问题和卡点

1. **框架选型的决策权重**：作为产品人，判断「用 LangChain 还是 LangGraph」的实际决策因素是什么？团队技术栈？生态成熟度？还是产品复杂度？有没有一个简单的「决策树」可以参考？
2. **无代码产品人如何「感受」框架差异**：不写代码的情况下，能否通过试用不同框架构建的 Demo 产品来理解它们的差异？比如找一个 LangChain 构建的问答机器人和一个 LangGraph 构建的多步 Agent，对比交互体验——这比读文档更直观。
