### 关键概念

<!-- 用自己的话记录 2-3 个核心概念 -->

1. **MCP 是什么？—— AI 的「USB 协议」**  
   MCP（Model Context Protocol）是 Anthropic 于 2024 年 11 月发布的开放标准协议。它的核心思路是：给 AI 模型定义一套统一的「插口」规范，任何外部工具或数据源只要遵循这个规范，就能被 AI 模型即插即用地调用。类比：USB 协议出现之前，每个外设需要自己的驱动；USB 出现后，所有设备用同一个接口。MCP 之于 AI 工具生态，就是 USB 之于硬件外设生态。

2. **MCP 的三层架构：Host → Client → Server**  
   - **Host（宿主）**：运行 AI 模型的应用（如 Claude Desktop、IDE 插件），是「使用者」  
   - **Client（客户端）**：嵌在 Host 里的连接器，负责与 Server 通信  
   - **Server（服务端）**：暴露具体能力的程序（如文件系统 Server、数据库 Server、GitHub Server）  
   一句话理解：Host 是「老板」，Client 是「助理」，Server 是「供应商」——老板告诉助理要什么，助理去找供应商拿。

3. **MCP 的三大「原语」：Resources、Tools、Prompts**  
   - **Resources（资源）**：Server 提供的「只读数据」——比如一个文件列表 Server 暴露 `file://documents/report.pdf`  
   - **Tools（工具）**：Server 提供的「可执行操作」——比如一个 GitHub Server 暴露 `create_issue`、`search_code`  
   - **Prompts（提示模板）**：Server 提供的「预制提示词」——比如一个 SQL Server 暴露 `"用自然语言解释这个查询"` 模板  
   产品视角：这相当于定义了 AI 插件的「功能菜单」——产品经理设计插件时，就是在规划这三个维度的能力清单。 

### 产品视角

> 从产品研究方向思考：这个概念对应什么产品场景？用户是谁？解决什么问题？

| 概念 | 产品场景 | 目标用户 | 解决什么核心问题 |
|------|----------|----------|-----------------|
| MCP 协议标准化 | AI 产品接入第三方工具/数据源 | AI 产品经理、开发者 | 以前接入每个外部服务都要写定制代码 → MCP 让「接入」变成「配置」 |
| MCP Server 市场 | 类似 App Store 的「AI 能力插件市场」 | AI 终端用户、企业 | 用户可以像安装 app 一样给 AI 助手安装新能力（如「加上 GitHub 能力」「加上飞书能力」） |
| MCP in Web3 | AI Agent 调用链上数据/执行链上操作 | Web3 用户、DApp 开发者 | 让 AI 助手能实时查询链上余额、gas 费、NFT 持有情况，甚至帮用户发起交易 |

**延伸思考**：
- 如果 OpenSea 做一个 MCP Server，暴露「查询 NFT 地板价」「查看收藏品持有分布」资源，你的 AI 助手就能在你的聊天窗口里帮你做 NFT 研究——不需要打开 OpenSea 网站。
- MCP 降低了 AI 产品间的「数据孤岛」：以前每个 AI 产品各自封闭，MCP 让不同产品的数据可以互通——前提是大家都遵循这个协议。这对产品的护城河策略是一个挑战：开放 vs 封闭，如何取舍？

## 问题和卡点

<!-- 记录今天遇到的困惑、不确定的地方 -->

1. **MCP 和 Function Calling 到底有什么本质区别？**  
   我理解 Function Calling 是模型直接调用一个预先定义好的函数/API——开发者需要在代码里手动实现每个 function。而 MCP 把这些 function 封装成标准化的 Server，让它们可以被任何支持 MCP 的 Host 复用。但我想确认：在实际产品开发中，选择用 Function Calling 还是 MCP 的决策边界是什么？

2. **如果我想做一个 AI × Web3 产品，MCP Server 由谁开发？**  
   是产品团队自己开发（需要技术能力）、用开源的现成 Server、还是未来会出现「MCP Server 市场」让第三方开发者来提供？对无代码的产品经理，我的角色是什么——定义需求然后找工程师实现？还是可以直接用现成的工具拼接？
