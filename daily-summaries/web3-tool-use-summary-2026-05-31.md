## 一、章节核心定义

Web3 Tool Use 是把 RPC 调用、合约读取、交易生成、钱包确认、区块浏览器查询和 DeFi 操作，封装成 AI Agent 可以调用的结构化工具的过程。
真正难的不是能调用，而是权限、参数、模拟和日志。

---

## 二、为什么要学这个

核心问题：如果一个 AI 助手说我帮你操作区块链，它到底需要具备什么能力、面临什么风险？

三个关键 insight：
1. AI Agent 不能只靠文本——需要工具读取链上状态、解释合约、估算 gas、生成交易、请求签名、查询交易结果
2. Web3 工具风险等级远高于普通查询工具——读错误导判断，写错改变资产和权限
3. 核心设计原则——每次链上动作都要有结构化输入、明确权限和可审计结果

---

## 三、第一性原理

模型可以选择工具，但工具必须用确定性边界限制模型。

三个基石原则：
- 读写分离：读取链上状态 vs 发送交易 = 不同工具、不同权限
- 参数结构化：chain_id、contract_address、method、args、value、slippage 不能埋在自然语言里
- 日志不可省：每次调用记录输入、输出、时间、来源、错误

核心安全理念：不要让 Agent 直接拼接任意 calldata 或调用任意地址。把危险能力封装成受限接口。

## 四、知识节点（共8个）

### 4.1 RPC Tool（难度：初级）
- 功能：读取链状态、查询区块、估算 gas、获取日志、广播交易
- 安全原则：只读开放，写入拆出，不与查询混在万能 RPC 工具里
- 返回标准：必须包含 chain_id、RPC provider、block_number、method、result、error

### 4.2 Contract Read（难度：初级）
- 功能：调用 view/pure 函数，不改变链上状态
- 典型用途：余额、配置、owner、allowance、pool 状态、价格、nonce、是否 paused
- 陷阱提醒：读错网络、读错合约、ABI 不匹配、RPC 数据滞后 -> 模型基于错误事实生成建议
- 产品视角：这是 Agent 最常用且相对最低风险的 Web3 工具类型

### 4.3 Contract Write（难度：高级）
- 功能：改变链上状态的交易执行
- 必经检查清单（8项）：chain_id + 目标合约地址、ABI method + args、value 和 token 变化预估、gas 估算、simulation 结果、policy 检查、用户或 Smart Account 授权、交易哈希和回执追踪
- 安全策略：白名单合约 + 白名单方法 + 额度策略

### 4.4 Wallet Tool（难度：高级）
- 功能：连接账户、请求签名、生成交易、管理授权、返回确认结果
- 最敏感边界：必须拆分连接/签名消息/发送交易/授权 token/撤销授权
- AI 生成的交易草稿不能绕过钱包确认，高风险动作 -> 人工/Smart Account policy/多签

### 4.5 Explorer Tool（难度：初级）
- 功能：查询交易、合约源码、event、token transfer、地址历史
- 核心价值：提供可验证证据

### 4.6 DeFi Tool（难度：高级）
- 功能：封装 swap、借贷、授权、仓位查询、清算风险等 DeFi 操作
- 必要约束（7项）：协议白名单、最大交易额、最大滑点、价格来源、simulation、allowance 检查、人工确认或 session key policy
- 禁止：不给 Agent 一个帮我调用任意 DeFi 协议的通用工具

### 4.7 Tool Permission（难度：高级）
- 功能：定义 Agent 能调用哪些工具、条件、参数范围
- 分层权限模型（L0~L4）：
  L0 查询余额 -> 自动允许
  L1 生成交易草稿 -> 自动允许
  L2 小额白名单支付 -> Session key 允许
  L3 大额转账或授权 -> 必须人工确认
  L4 任意合约调用 -> 默认禁止
- 权限维度：工具、合约、方法、金额、时间、频率、用户确认级别

### 4.8 Tool Log（难度：中级）
- 功能：Agent 行为可审计的基础
- 必须记录字段：用户目标、工具名、输入参数、输出结果、错误、时间、链 ID、区块号、交易哈希、确认人、policy 判断
- 产品价值：当 Agent 出错时，日志能回答 模型看到了什么 -> 调用了什么 -> 工具返回了什么 -> 系统拦截了吗 -> 用户确认了什么

## 五、在 AI x Web3 中的位置

Web3 Tool Use 是从 AI 能解释链上信息 走向 AI 能参与链上执行 的关键层。

上下游关系：
上游：Chain-aware Context（链感知上下文）-> 提供链上数据
本层：Web3 Tool Use（本章）-> 封装执行能力
下游：Agent Workflow + Agent Wallet -> 编排执行流程、管理资产

如果工具只读 -> 风险主要是解释错误
如果工具能写链 -> 风险进入资产和权限层
越靠近执行，越需要 policy、simulation、human check 和日志

## 六、最小实践（5步动手设计）

1. 只读工具设计：设计一个读取某地址在某链上 ETH 余额的工具 schema
2. 合约读取工具设计：设计读取 ERC-20 allowance 的工具
3. 交易草稿工具设计：设计生成 ERC-20 approve calldata 但不发送交易的工具
4. 权限规则设计：定义只允许特定 token、特定 spender、最大额度的写交易权限
5. 日志字段设计：为每个工具定义输入 schema、输出字段、错误类型和日志字段

重点不是实现，而是把读、写、签名、确认、记录五个边界分清楚。

## 九、产品方向学习建议

对于 AI 新手 + Web3 新手 + 无代码 + 产品研究方向 的初学者：

### 当日学习重点（3个核心概念）
1. 读写分离 — Web3 工具安全设计的铁律
2. 权限分层 — L0~L4 五级权限模型，产品经理可直接套用
3. 日志不可省 — 可审计性是 AI Agent 产品合规的基础

### 当日思考题
- 如果你设计一个 AI 链上助手产品，你会开放哪些工具给用户？每个工具的权限边界在哪？
- 对比传统 SaaS 产品的权限系统，Web3 Tool Use 的权限设计有哪些本质区别？
- 假设一个用户投诉 AI 帮我转错了钱，日志应该能回答哪些问题？

### 一个关键产品洞察
Web3 Tool Use 的本质不是技术问题，而是信任边界设计问题。它连接 AI 的不确定性和区块链的不可篡改性，中间的每一层工具封装都是一道安全阀门。
